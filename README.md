# Лабораторная работа №3: Сравнение производительности коллекций

## Описание проекта

Данный проект предназначен для практического сравнения производительности двух популярных реализаций интерфейса `List` в Java: `ArrayList` и `LinkedList`.

Программа выполняет высокоточные замеры времени (с использованием `System.nanoTime()`) для 9 основных сценариев: операций добавления, чтения и удаления в начале, в середине и в конце списка.

## Технологии

*   Java 24 (Corretto)
*   Maven
*   JUnit 5

## Как запустить проект

2.  **Запуск:**
    *   **Способ 1 (Рекомендуемый):** Открыть проект в любой IDE, поддерживающей Maven (например, IntelliJ IDEA), и запустить метод `main` в классе `Main.java`.
    *   **Способ 2 (Командная строка):** Выполнить команду Maven, которая скомпилирует и запустит программу.
        ```bash
        mvn compile exec:java -Dexec.mainClass="com.sergey.Main"
        ```

## Результаты тестов и их анализ

Ниже представлены реальные результаты выполнения тестов. Время указано в миллисекундах (ms).

### Тестирование ArrayList

| Метод                     | Количество        | Время (ms) |
| ------------------------- | ----------------- | ---------- |
| Добавление в конец        | 100000            | 2,925      |
| Добавление в середину     | 1000              | 7,614      |
| Добавление в начало       | 1000              | 7,563      |
| Чтение с начала           | 100000            | 1,768      |
| Чтение из середины        | 100000            | 0,510      |
| Чтение с конца            | 100000            | 0,424      |
| Удаление с конца          | 1000              | 0,100      |
| Удаление из середины      | 1000              | 4,338      |
| Удаление с начала         | 1000              | 8,233      |

**Анализ:**
*   **`get` (чтение):** `ArrayList` демонстрирует практически мгновенный доступ к элементам в любой части списка (0.4-1.7 мс), что подтверждает его сложность `O(1)`.
*   **Операции в конце:** Удаление с конца (`0.100 мс`) и добавление в конец (`2.925 мс`) очень эффективны.
*   **Операции в начале/середине:** Это худший сценарий (`O(n)`). Даже для 1,000 операций время измеряется в миллисекундах, так как `ArrayList` вынужден физически сдвигать огромное количество элементов в памяти.

### Тестирование LinkedList

| Метод                     | Количество        | Время (ms) |
| ------------------------- | ----------------- | ---------- |
| Добавление в конец        | 100000            | 1,814      |
| Добавление в середину     | 1000              | 70,020     |
| Добавление в начало       | 1000              | 0,025      |
| Чтение с начала           | 100000            | 2,453      |
| Чтение из середины        | 100000            | 5986,698   |
| Чтение с конца            | 100000            | 2,308      |
| Удаление с конца          | 1000              | 0,127      |
| Удаление из середины      | 1000              | 55,902     |
| Удаление с начала         | 1000              | 0,037      |

**Анализ:**
*   **Операции в начале/конце:** `LinkedList` здесь абсолютный чемпион. Добавление и удаление занимают сотые доли миллисекунды (`0.025-0.127 мс`), так как требуют только переназначения пары ссылок (`O(1)`).
*   **`get` (чтение из середины):** Это его ахиллесова пята (`O(n)`). Поиск элемента в середине занял **почти 6 секунд**, так как `LinkedList` последовательно прошел по десяткам тысяч ссылок.
*   **Операции в середине:** Также очень медленные (`55-70 мс` для 1,000 операций). Основное время тратится не на саму вставку/удаление, а на **поиск** нужного элемента в середине списка.

## Вывод

Проведенные тесты наглядно демонстрируют фундаментальные различия в устройстве `ArrayList` и `LinkedList`.

*   **`ArrayList`** — идеальный выбор, когда основной операцией является частое чтение данных по индексу.
*   **`LinkedList`** — незаменим, когда требуется множество операций вставки или удаления в начало/конец списка (например, при реализации очередей или стеков).
